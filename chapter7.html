chapter 7 - understanding class inheritance


Inheritance
  • allows dev to create general classes that can be used as a foundation for more specific classes
  • typical animal example - general class Animal.java, specific classes Dog, Cat and Horse.java
  • general class shares common elements - colour, weight, age, etc. and method to do common activities such as eat, move and rest
  • the purpose of this general class is to prevent code repetition in each of the more specific animal type classes
  • benefits of this are greater than less code, also makes bug resolution and code maintenance easier, new code can be added to the general class and be readily accessible to all child classes
  • duplicated code also increases the risk that the same code in different classes performs differently as it goes through the dev process, unless each block of code is updated exactly the same
  • general classes also allow for polymorphism - in the above instance, a Dog object can be referred to and treated as an Animal object
  • the base class is known as the superclass, which is extended by child or sub classes, which in turn inherit all of the nonprivate variables and methods
  • a class can only extend 1 superclass, however, a class can inherit from one class which inherits from another and so on
  • once a class inherits a superclass, it gains non-private methods and variables, and adds to that any specific variables in its own class OR remain simply as it is
  • X extends Y is the same as saying Y inherits from X

Overrriding Methods
  • used when methods must be slightly tailored for a subclass
  • eg, if a fish object was created, it does not move it swims...
  • the initial method can be overridden by declaring another method in the Fish class with the same name and adding the annotation for @override
  • when the move method is then called for a Fish object, the overridden method in the Fish class is executed
  • note that for an overrride, the method signatures must be identical
  • the use of the keyword super can be used to access an overridden method
  • an example of this would be a horse object, as horses need to stand before resting, a typical rest method would move the horse into a standing position, before callinging the overridden rest.super() method - again parameters must match exactly
  • therefore it is possible to have additional functionality in an overriding method, before the initial overridden class is called

Abstract Classes
  • concrete classes are regular classes that can be instantiated
  • abstract classes cannot be instantiated, and may contain abstract methods - methods that are not implemented
  • abstract methods must have a valid method signature, and must be overridden and implemented in a concrete class which extends an abstract class
  • an abstract class can extend another abstract class, without implementing its methods
  • it is also possible for an abstract class to have variables inherited by any concrete class which extends the abstract class
  • an example is a program for a music player - which can be used by a mp3 player, a cd player, a computer, a phone, etc etc
  • an abstract class for this program may contain methods such as play, stop and change volume
  • not only is the abstract keyword used in the class name, but in any abstract methods within the class
 public abstract class MusicPlayer {
    public abstract void play(){ };
    public abstract void stop() { };
    public void changeColume() {
    // change volume level;
    }
}
  • every abstract method MUST be implemented by the first concrete subclass - these methods cannot be shared between subclasses, as for example, the mp3 player will first need to decode the file and then play, whilst the cd player will need to start reading the disk and then play

Interfaces
  • used to define a required set of functionalities for the classes that implement the interface
  • different from extending a base class, as a class is free to implement as many interfaces as needed
  • similar to an abstract class with all abstract methods
  • similar to abstract classes, when a concrete class implements an interface class, all the methods contained in the interface must be implemented
  • an abstract class that implements an interface class must not implement all methods contained in the interface, but the first concrete class which extends this abstract class must
  • interfaces are used to create a standard public interface for similar items, which makes thee code more modular
  • when more than one interface is being implemented, they are separated with a comma delimited list
  • the use of an interface ensures that any class implementing it will have the same public methods as other classes - an example is a Phone interface class used with a CellPhone, LandlinePhone and VideoPhone object. The VideoPhone class may also implement an interface for a VideoPlayer

Advanced Concepts of Inheritance
  • every class inherits from the base class Object.java - this is implied and does not have to be specifically accessed using the extends keyword
  • this means that, to start off with, we have at least 1 inherited class
  • if we extend the animal class in the dog class, we inherit not only the animal class but also the object class
  • if a class overrrides a method in another, the overridden class is passed down
  • an inheritance chain is created between multiple classes, and can be continued as long as it makes computing sense in the application
  • an interface can also extend multiple interface classes using the extends keyword and a comma delimited list for more than one interface

Encapsulation
  • encapsulation is the concept of storing data together with methods that operate on that data
  • the code is stored in an object
  • encapsulation allows for data and method hiding - a.k.a. information hiding
  • this makes it possible to expose a public interface while hiding the implementation details

Good design with Encapsulation
  • one of the benefits of O-O programming, is the association between methods and variables in the code, other languages which do not have this relationship make finding the source of bugs a lot harder
  • encapsulation attempts to solve such problems, code is created in an object container (classes), making it modular
  • a well encapsulated class has a single and clear purpose - and contain the methods and variables needed to fufill this purpose
  • the purpose of a well encapsulated class should be clearly defined
  • some say encapsulation is only about the grouping of related methods and variables, for the OCA exam, it also involves the information hiding of the implementation details
Access Modifiers
  • the keywords that define access
  • allow a dev to determine which parts of a class are viewable and accessible outside of the class
  • 3 defined modifiers, as well as a default (blank)
  • can also be used when defining classes, but not in exam
private
  • most restrictive, and most commonly used
  • any methods or variables declared as private, can only be accessed within that class
  • subclasses CANNOT access private methods in superclasses
default (package-private)
  • the second most restrictive
  • allows access only within the same package
  • no declaration (left blank)
protected
  • 3rd most restricted
  • same as default but allows subclasses outside of a package access
  • must be in the same package, or be a subclass of a class within the same package
public
  • least restrictive
  • 2nd most common
  • provides no restriction - any method can access a public method or variable
  • a.k.a world access
Information Hiding
  • hiding the implementation details of a class
  • achieved through access modifiers
  • this provides the dev control as to how data is accessed, and therefore reduces the risk of bugs as outside objects can be controlled
  • often, a class will contain more internal methods to process the data, and a few public getters and setters to allow access from outside to provate variables
  • general design involved assuming private first, which helps reduce bugs, and only change if necessary
Exposing Object functionality
  • any methods within a well incapsulated class that are used to manipulate the data should be private
  • public methods within a class can also be thought of as interfaces, which allow access to the class functionality
  • external objects have no knowledge of the class implementation, only of the public getters and setters
  • getters provide read access, and setters wrtie access to provate variables
  • getters should start with a lowercase getSomethingSomething, as setters start with setSomethingSomething
  • the one exception to the above convention is when dealing with boolean values - isSomething would be used
