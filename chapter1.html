chapter 1 - packaging, compiling and interpreting

Packages
  • most common way of grouping and organising related classes and interfaces
  • though of as containers for classes & define where that will be located in the hierarchical directory structure
  • decreases likelihood of classes colliding
  • promotes code reuse, maintainability, O-O encapsulation and modularity
  • also allows for classes across a normal or large piece of software to have the same name - as long as they are in separate packages

Import Statements
  • java.lang is imported into each java class by default
  • package com.example.package at the top of a source file will locate the class within a package - only association with 0 or 1 package is possible
  • import and then package name to important the components of any package
  • use * to import all classes of a package, or .className to import a specific class
  • convention dictates package name to be a reverse domain name com.example.package but can also be package example.package || package example_package
  • the package name equates to a directory structure path/example/package or path/example?package
  • package names java and java are reserved
  • individual names with a package name to be separated with an underscore
  • one import statement per package
  • import keyword followed by path to package directory
  • some useful packages to know:
    • basic graphics and images - import java.awt.*
    • lightweight GUI components - import java.swing.*
    • data streams - import java.io.*
    • networking application - import java.net.*
    • collections framework, event model & date.time facilities - import java.util.*
    • core java classes and interfaces - import java.lang.* (core package, imported by default)
  • for maintenance purposes it is better to call classes specifically rather than using *

Static Imports
  • allow specific package elements to be imported
  • for example import static java.util.Locale.ITALY imports the local it_IT
  • sout calls which use ITALY would print this information, and this information only
  • import java.utils.* would allow the system to also print out the locals for GERMANY (de_DE) etc.

Understanding Packaged Derived Classes
  • total of 209 packages in Java 7
  • most important are:
    • utilities
    • basic input/ output
    • networking API
    • Abstract Window Toolkit API
    • Swing API

Java Utilities API
  • java.util
  • classes are arranged into categories such as Framework, Date & Time, Internationalisation, and others
  • THe Java collections framework contains fundamental data structures necessary to build Java Applications
  • List - ArrayList, LinkedList, Vector - all data structures based on positional access Map - Hashmap, Hashtable, LinkedHashMap, TreeMap - structures that map keys to values Set - Hashset, LinkedHashSet, TreeSet - data structures based on element uniqueness Queue Interfaces - PriorityQueue - FIFO manner usually used in queues, priority queues order elements accoring to a supplied comparator
  • the comparator interface is there to assist when ordering is not natural
  • not to be confused with the comparator in the java.lang package, which is to sort objects by their natural ordering
  • Data, Calender and TimeZone classes are also in .util
  • Locale class used for geographical regions
  • Currency class represents curriencies
  • Random class includes a random number generator
  • StringTokenizer class can split a string into tokens
  • there are many more, perhaps used onn the job. Diagram on page 11 shows various possibilities
NB packages can also contain subpackages to further help with the parent package. .util.regex includes regular expressions. Subclasses tend to have unique fuinctionality relating to the parent class

Java basic input/ output API
  • both combined in Java.io
  • provides functionality for general system input and output
  • includes data streams, serialisation, file system
  • subclasses include InputStream and OutputStream for the bytestream class
  • characterStream subclasses include Reader and Writer classes
  • other important classses include the File, FileDescriptor, FilenameFilter and RandomAccessFile which all allow foe the creation, manipulation and removal of files

Java Networking API
  • contained in the package java.net
  • allows for the creation of network based applications
  • Classes include Socket (client side), ServerSocket (server side), URL,m Inet4Address, Inet6Address
Java Abstract Window Toolkit
  • java.awt
  • can create heavyweight components in regards to UX
  • and associated graphics and images
  • has since been superseded with SWING, but certain pieces of .awt are still used
  • eg the FocusSubsystem which allows navigation between components
Java Swing API
  • javax.swing
  • functionalight for lightweight containers and components
  • more sophisticated set of UI components than AWT
  • previous elements in AWT, eg button, are simply preceded with a J in SWING , JButton
  • javax.swing.* imports all inc buttons, tables, dialog boxes
  • swing.text for textual elements only
  • swing.platf.* to import elements to allow for the implementation and configuration of basic pluggable look and feel support (allowing for universal style changes of the GUI's components. Also included are tooltips, accessibility functionality, event model, tables, trees, etc.
  • swing.event.* for event listeners and adapters
  • other examples include JFrame, JPanel, JToolTip - see p 15
  • swing takes advantage of the MVC (Modal-View-Controller) architecture - not needed for exam
  • note that SWING is to be replaced with FX javafx

Compile & Interpret Java Code
  • JDK has a number of tools available, look inside package for range of functions
  • javac used to cimpile source code into bytecode
  • javac [options] [filename]

-d option
  • allows user to determine in which directory the compiled code will do
[present working directory]\classes\com\ocajexam\tutorial\GreetingsUniverse.class

-classpath option
  • used to tell the JVM where user-defined classes and packages are located
  • can also be shortened to -cp
  • -d can still be used as follows, to ensure that the bytecode goes into the classes directory
javac -d classes -cp 3rdPartyCode\classes;. GreetingsUniverse.java

  • no need to use -cp if the files are in the present working directory
  • also no need if the classpath is defined in the environmental variable CLASSPATH
  • NB that in windows systems, paths are with \; and delimited with semi-colons, not /: as with (POSIX) UNIX, delimited with colons
  • for the exam, it is important to know the different types of switches: also known as command-line parameters, command-line switches, options and flags

Interpreting Java code
  • java [filename] on compiled code
  • takes byte code and runs program
  • on a windows machine, javaw.exe [filename] can be used to open the program and close the console
  • on a linux machine, java [filename] & can be used to run the program in the background
  • may be required to declare where certain classes are kept before running the program, again this can be accomplished with the -cp or -classpath option
  • in the event that all the required classes are included in 1 package - as they should be - it is possible to point the interpreter at the base folder of the package (com.coderwurst.project) to include all classes under this package
  • when interpreting code, the -D option allows properties within class files to be updated

Interpreter Version
  • - version can be used with the interpreter to return the version of the JVM and exit
  • a number of additional arguments can also be used with this command, with varying results
