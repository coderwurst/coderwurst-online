chapter 10 - working with classes and their relationships

relationships

  • often described in terms of association and composition
  • 2 general terms for object relationships, which encompass 4 types of relationship descriptors; direct association, composition association, aggregation association and temporary association

Class Compositions and Associations
  • relationships are formed between 2 objects when 1 contains a reference to another
  • the reference is often stored as an instance variable
  • this reference may be uni or bi directional
  • association is a relationship between 2 objects, where neither one directly depends on the other for its logical meaning
  • this means, in an association relationship the link between the 2 objects could be disconnected, and they would still have logical meaning - weak relationship
  • objects in an association relationship have no dependance on each other for the management of their life cycle- the existence of one is not dependant on the existence of another
  • think of a CarFactory and a CarFrame example, a factory can produce cars with numerous types of frames
  • a composition relationship is a stronger type of relationship - one object is composed of another
  • an object may be composed of multiple objects, not just between 2
  • if the connection is lost, these objects have no longer any logical meaning
  • this tends to be a more abstract relationship - think of a Car and CarStatus objects, one cannot exist without the other
  • objects that control the life cycle of another object tend to have a composite relationship
  • usually some level of interpretation is needed to determine the type of relationship between 2 objects
  • a composition relationship between 2 objects will always be responsible for an object's life cycle, objects in association tend to make more sense by themselves than objects of composition

Class Relationships
1. Direct Association
  • "has" a relationship
  • basic level of association (default), represents navigability
  • weak association, generalised into an association
  • no life cycle responsibility and each object is conceptually independent
  • think of Truck and Trailer Object - weakly associated, both could be used without the other
  • represented in Unified Modeling Language (UML) as as a single arrow

2. Composition Association
  • one object is composed of 1 or more objects
  • strong relationship, generalised as a composition
  • the internal object will make only conceptual sense while stored in the containing object
  • this relationship tends to reflect ownership - Object A is composed of Object B
  • Think of a Tyre and RubberStrips example
  • the containining object (Type) is responsible for managing the life cycle of the internal object, this responsibility can also be passed to another object
  • a reference from the containing object must always be held to the internal object, otherwise it will be garbage collected

3. Aggregation Association
  • 1 object is a part of another object - "part of" the whole relationship
  • one object is part of the other, each object can maintain its own meaning independantly if the relationship is lost
  • neither object depends on the other for its existence
  • no need for life cycle management
  • weak relationship that can be generalised as an association
  • think of a Motorcycle and a windshield object - weakly associated as both could be used without the other and still maintain the initial purpose
  • but the windshild is "part of" the motorcycle

4. Temporary Association
  • also known as a dependency
  • typically local variables are temporary associations, but also a return value or a method parameter
  • although Object A will depend on Object B, it will not persist for the entire life cycle of the object
  • For example, a Car Object may have a mthod to Start Engine, which requres a Key Object as a parameter

Multiplicities
  • every relationship has one
  • represent the number of objects that are part fo the relationship
  • one to one, one to many & many to many
One-to-One Multiplicity
  • basic relationship where one object contains a reference to one other object
  • Eg. Motorcycle Object to Engine Object
  • all 4 of the above objects may have a 1-1 relationship
One-to-Many Multiplicity
  • one object contains references to a group of objects
  • the multiple objects are usually stored as an array or a collection
  • all 4 of the above relationship types may be 1-many
  • Eg. Car has 4 Tyre Objects, the type is 'part of' the car - aggregation association
Many-to-Many Multiplicity
  • only possible for aggregation associations, direct and temporary
  • i.e. not for composition, due to the fact that is many object have a relationship with an object, it is impossible for any individual object to control the life cycle of another object
  • EG. a traffic simulator program; many Car Objects, with further references to many TrafficLight Objects. The relationship between the Car and the Traffic light is weak, and all traffic light objects are shared with all the cars
  • a many to many association fors not have to have an equal number of objects on each side of the relationship
Association Navigation
  • used to describe the direction in which a relationship can be travelled
  • an object within another object is said to be navigable is the containing object has methods for accessing the inner object
  • most relationships are navigable in 1 direction, but if both objects contain references to the other, it is possible to have bi-directional navigational relationship
  • getters and setters are used here

Class Compositions and Associations in Practice
Examples of Class Association Relationships
  • one-to-one class association Truck - Trailer example
    • Truck object, contains reference to the trailer object and a method to set the trailer object
    • the trailer object is a single variable (not in an array or a collection) and is therefore a 1-1 relationship
    • direct association; truck object not responsible for trailer life cycle, and we can say that the truck 'has a' trailer object
    • this type of relationship can be generalized to an association relationship
  • one-to-many class association - Car - Wheel example
    • aggregation association as the wheel is 'part of' the car object
    • one to many relationship as 1 car has 4 wheels, wheel objects stored in an array via the setWheels method
    • weak relationship and no life cycle dependencies, therefore this can be generalised as a association relationship
  • many-to-many class association - Car - Traffic Light example
    • more complex example than the first 2
    • groups of car objects, groups of traffic light objects
    • 3rd class in this example is the traffic simulator - key to understanding the relationships
    • TrafficLight object contains an id reference
    • Car Oject contains a reference to all the traffic lights
    • Simulator instantiates a TrafficSimulator Object, which contains an array of Car objects and an array of Traffic Light objects. Each TrafficLight object is assigned a unique id, then the car objects are created and this array of TrafficLight objects is passed in
    • this can be said to be a direct association, as each Car have an array containing the TrafficLight objects
Examples of Class Composition Relationships
  • one-to-one composition - Tyre and TyreAirPressure
    • the tyre object is 'composed of' the TyreAirPressure obect - composition association
    • strong relationship
    • Tyre is reposnsible for the life cycle of the TyreAirPressure Object
    • if the Type were destroyed, the AirPressure would also be gone
  • one-to-many composition - CarComputer - SensorStatus obect
    • the CarComputer object is composed of an array of 5 sensors - one-to-many relationaship
    • the relationship is a composition association
    • strong relationship, SensorStatus depends on the CarComputer to survive
Examples of Association Navigation
  • PinStripe object 'composed of' a Colour object
  • PinStripe object contains a method to get the colour object
  • therefore, any object that has access to the PinStripe object could use this method to return a colour
  • this relationship allows navigation in 1 direction only
